## 스프링 입문 - 코드로 배우는 스프링 부트, 웹 MVC, DB 접근 기술

> 2026.03.22 완강

간단한 웹 애플리케이션을 개발해보면서 배운다
- 스프링 프로젝트 생성
- 스프링부트로 웹 서버 실행
- 회원 도메인 개발
- 웹 MVC 개발
- DB 연동 - JDBC, JPA, 스프링 데이터 JPA
- 테스트케이스 작성

스프링 기술 그 자체에 매몰되기보다는, 어떻게 사용해야하는지에 초점. 이 강의에서는 마이너한 기술 쳐내고 핵심만 다룰 것

---

컨트롤러가 멤버 서비스에 의존하는걸 스프링스럽게?

스프링이 뜰 때 스프링 컨테이너라는 통이 생긴다  
컨트롤러 어노테이션이 있으면, 이 멤버 컨트롤러 객체를 생성을 해서 스프링이 통에 넣어둔다  
이걸 스프링 컨테이너에서 스프링 빈이 관리된다고 표현함

근데 매번 객체를 new 해서 쓰는건 비효율적이다.  
필요한 의존성을 직접 생성해서 쓰지 말고, 스프링 컨테이너에서 받아서 쓰도록 해줘야 함

지금은 멤버컨트롤러만 멤버 서비스를 쓰지만, 다른 컨트롤러에서도 멤버 서비스가 필요해질 수 있다  
멤버 서비스에는 별다른 기능이 없다. 굳이 여러개의 인스턴스를 생성할 필요가 없음 -> 하나만 만들어두고 공용으로 쓰면 된다

`@Controller`, `@Service`, `@Repository` 모두 `@Component` 어노테이션의 특수한 형태!

컴포넌트 스캔?  
스프링이 뜰 때, 컴포넌트 관련 어노테이션이 있으면 전부 객체를 하나씩 만들어서 `스프링 컨테이너너`에 등록해둔다 (통에 넣어둔다)  
`@Autowired`는 연관관계를 설정해주는것 (컨트롤러가 서비스를 쓰고, 서비스가 리포지토리를 쓸 수 있도록)

컴포넌트 스캔의 범위?  
`@SpringBootApplication` 어노테이션 붙은 클래스가 있는 패키지를 포함해서, 그 하위에 있는 모든 패키지들을 스캔함

\+ 스프링 컨테이너에 스프링 빈을 등록할 때, 기본적으로 `싱글톤`으로 등록된다 (객체를 하나만 만들고 공유)  
\+ 따라서 같은 스프링 빈이라면 모두 같은 인스턴스인 셈이다 (싱글톤이 아니게 등록할 수도 있지만, 웬만하면 싱글톤만 쓴다)

스프링을 쓰면, 웬만한 건 전부 스프링 빈으로 등록해서 쓰는게 좋다

---

JPA를 쓰려면, 데이터를 저장하거나 변경할 때 항상 트랜잭션이 있어야 한다  
서비스 계층에 `@Transactional` 붙여주기! (엄밀하게는 DB 데이터를 수정하는 메서드)

JPA는 모든 데이터 변경이 트랜젝션 안에서 실행되어야 한다!!

---

`JpaRepository`를 extend하는 인터페이스를 만들어두면,  
`스프링 데이터 JPA`에서 해당 인터페이스의 구현체를 자동으로 만들어서 스프링 빈으로 등록해준다!

findByName, findByNameAndId, ...  
And, Or 등등 이름 짓는 규칙이 있다

메서드 이름이랑 리턴 타입, 파라미터 같은 정보들을 다 리플렉션으로 읽어들여서 `스프링 데이터 JPA가 알아서 만들어준다`

인터페이스 이름만으로 단순한 80%의 리포지토리 메서드는 개발이 끝난다  
그럼 나머지 복잡한 20%는??

스프링 데이터 JPA + 복잡한 동적 쿼리는 `QueryDSL 라이브러리`  
위 조합으로도 해결하기 어려운 쿼리는 JPA가 제공하는 `네이티브 쿼리(생짜 SQL)`를 사용하거나, `JdbcTemplate`를 사용하면 된다

---

스프링에서 AOP가 어떻게 동작하나?

AOP가 붙으면, 가짜(프록시) 스프링 빈을 만든다  
그 프록시 스프링 빈에서 이런저런 작업들을 하고, `joinPoint.proceed()`가 호출될 때 진짜를 실행한다
