## 인터넷 네트워크

왜 네트워크부터 시작?  
결국 웹이나 HTTP도 전부 인터넷 네트워크 망 위에서 동작하기 때문

인터넷에서 컴퓨터 둘은 어떻게 통신할까?  
컴퓨터 두 대가 바로 옆에 붙어있다면, 그냥 케이블을 연결해서 메시지를 주고받을 수 있을 것  
그런데 두 컴퓨터가 멀리 떨어져 있다면? 즉, `사이에 인터넷이 있다면?`  
`인터넷 망`을 통해 메시지를 주고받아야 한다

## IP (인터넷 프로토콜)

인터넷에는 수많은 노드가 있다. 그중 누군가를 콕 집어서 메시지를 보내려면 최소한의 규칙이 있어야 할 것  
그게 바로 `IP 주소`  

먼저 메시지를 보내려는 클라이언트가 `IP 주소`를 가져야 한다  
그리고 메시지를 받아야 하는 서버도 `IP 주소`를 가져야 한다  

- IP 프로토콜(인터넷 프로토콜)의 역할: `지정한 IP 주소`에 데이터를 전달할 수 있도록 해 줌  
    - 패킷(Packet)이라는 통신 단위로 데이터를 전달
    - IP 패킷에는 `출발지 IP`, `목적지 IP`가 있어야 한다 (주소를 적듯)
    - (출발지 주소, 목적지 주소, 데이터) 를 담은 IP 패킷을 인터넷 망에 던진다
    - 인터넷 망에 있는 서버들은 다 IP 규약을 따르고 있기 때문에, `목적지 IP`를 향해 이 패킷을 전달해줌

- IP 프로토콜의 한계
    - 비연결성
        - 패킷을 받을 대상이 없거나, 서비스 불능 상태여도 패킷을 전송함
    - 비신뢰성
        - 중간에 패킷이 사라지면?
        - 패킷이 순서대로 오지 않으면?
    - 프로그램 구분 불가
        - 같은 IP를 사용하는 서버에서 통신하는 애플리케이션이 두 개 이상이면?
            - 인터넷 게임도 하고, 음악도 듣는다면 두 서비스를 어떻게 구분?

## TCP, UDP

- 인터넷 프로토콜 스택의 4계층:
    - 애플리케이션 계층 (HTTP, FTP, ...)
    - 전송 계층 (TCP, UDP)
    - 인터넷 계층 (IP)
    - 네트워크 인터페이스 계층

1. 응용 프로그램에서 "Hello, World!" 메시지 생성
2. SOCKET 라이브러리를 통해 운영체제에게 전달됨
3. TCP 정보 생성, 메시지 데이터 포함
4. IP 패킷 생성, TCP 데이터 포함
    - -> 네트워크 인터페이스를 통해 전송됨

`패킷`은 Package + Bucket 합성어. 데이터를 담아서 보내기 위한 박스

- IP 패킷 안에 TCP와 관련된 정보가 추가로 들어간다
    - 출발지 PORT
    - 목적지 PORT
    - 전송 제어
    - 순서
    - 검증 정보
    - ...

> 이런 정보들을 통해, IP 프로토콜만으로는 해결이 안 됐던 문제들을 해결할 수 있다

- TCP(Transmission Control Protocol), `전송 제어 프로토콜`
    - 연결 지향적 (3-way handshake를 통해 연결을 먼저 맺고 데이터를 보낸다)
        1. SYN
        2. SYN + ACK
        3. ACK
        - 연결이 된 것을 확인한 뒤에 메시지를 전송한다 (3. ACK와 함께 데이터를 전송할 수도 있다)
        - 이때 `연결`은 물리적인 연결이 아니라 `논리적인 연결`이다
    - 데이터 전달 보증 (중간에 패킷이 누락되면 알 수 있다)
        - 데이터를 전송받고 나면 `데이터 잘 받았다`는 메시지를 보내도록 되어 있음
        - 데이터를 보냈는데 응답이 없다면 뭔가 문제가 생겼다는 것을 알 수 있다
    - 순서 보장 (패킷들 간의 순서를 보장해준다)
        - 중간에 누락된 패킷이 있다면 다시 받을 수 있음

- UDP(User Datagram Protocol), 사용자 데이터그램 프로토콜
    - IP 프로토콜과 거의 같다.
        - `PORT`, `체크섬` 정도만 추가된 것
        - 애플리케이션 계층에서 추가적인 작업이 필요하다
    - 하얀 도화지 비유 (기능이 거의 없음)
    - 연결지향 X, 전달 보증 X, 순서 보장 X
    - 단순한 대신 빠르다

> TCP는 `신뢰할 수 있는 프로토콜`
> 기본적으로는 TCP를 사용한다고 알고 있으면 됨
> 현재는 대부분의 애플리케이션에서 TCP 프로토콜을 사용함 (점유율이 90% 이상)
>
> 최근에는 UDP가 다시 각광을 받기 시작함. HTTP 3 같은 경우 UDP 기반

## PORT

IP 주소는 컴퓨터마다 하나씩 부여받는 것  
그런데 한번에 둘 이상 연결해야 한다면? (게임도 하고, 음악도 듣고)  
클라이언트 컴퓨터가 한번에 여러 개의 서버랑 통신해야 하는 경우, 어느 애플리케이션이랑 통신하기 위한 데이터인지 구분할 방법이 필요  

`IP`는 목적지 서버를 찾기 위한 것  
`PORT`는 그 서버 안에서 돌아가는 애플리케이션들을 구분하기 위한 것

IP는 `아파트`, PORT는 `몇 동 몇 호`

포트 번호는 `0 - 65535` 사이에서 할당 가능  
하지만, `0 - 1023` 까지는 사용하지 않는 것이 좋다 (기존에 잘 알려져있는 다른 서비스와 충돌할 가능성이 높음)

- 유명한 포트 예시:
    - 80: HTTP
    - 443: HTTPS
    - 22: SSH
    - 21, 20: FTP

## URI

URI == `Uniform Resource Identifier`  
`리소스를 식별하는 방법`

- URI? URL, URN?
    - “URI는 로케이터(Locator), 이름(Name), 또는 둘 다 추가로 분류될 수 있다”
    - https://www.ietf.org/rfc/rfc3986.txt (표준 스펙 정의)

- `URI` 라는 큰 개념이 있고, `URL`, `URN`은 하위 개념
    - URL 예시: `foo://example.com:8042/over/there?name=ferret#nose`
        - `foo` : scheme
        - `example.com:8042` : authority
        - `/over/there` : path
        - `?name=ferret` : query
        - `#nose` : fragment
    - URN 예시: `urn:example:animal:ferret:nose`
        - 이름을 부여하는 것
        - 그런데 이 이름만 가지고는 리소스를 찾기가 거의 불가능하다
        - 거의 안 쓴다. 이런게 있다 정도만 알아두기

URI라는 단어의 뜻
- Uniform: 리소스를 식별하는 `통일된` 방식
- Resource: 자원. 즉, URI로 식별할 수 있는 모든 것 (종류에 제한 없음)
- Identifier: 다른 항목과 구분하기 위해 필요한 정보

URL - Locator, 리소스가 있는 위치를 지정  
URN - Name, 리소스에 이름을 부여  
  
위치는 변할 수 있지만, 이름은 변하지 않는다  
`urn:isbn:8960777331` (어떤 책의 isbn URN)  
  
하지만 URN 이름만으로 실제 리소스를 찾을 수 있는 방법이 보편화되어있지 않다  
그래서 `URI와 URL을 크게 구분하지 않고 사용하기도 함`

- URL scheme
    - `scheme://[userinfo@]host[:port][/path][?query][#fragment]`
    - https://www.google.com:443/search?q=hello&hl=ko

- scheme
    - 주로 프로토콜이 들어간다
        - 프로토콜: 어떤 방식으로 자원에 접근할 것인가 하는 규칙
        - `http`, `https`, `ftp` 등등
    - `http`는 `80` 포트, `https`는 `443` 포트를 주로 사용함
    - `http`나 `https` 프로토콜의 경우, 포트는 생략 가능
        - https는 http에 보안 레이어가 추가된 것 (HTTP Secure)
- userinfo
    - URL에 사용자 정보를 포함해서 인증해야 할 때 사용
    - `거의 사용하지 않는다`
- host
    - `도메인 명` 또는 `IP 주소`
- port
    - 생략할 경우 `http`는 `80`, `https`는 `443`
- path
    - 리소스가 있는 경로
    - 주로 계층적 구조를 가지도록 설계함 (디렉토리처럼)
        - /home/file1.jpg
        - /members/100
        - /items/iphone17
- query
    - `Key-Value` 형태
    - `?`로 시작하고, `&`로 추가도 가능
        - `?keyA=valueA&keyB=valueB`
    - `query parameter`, `query string` 등으로 불린다
        - 웹 서버에 제공하는 파라미터라서 query parameter
        - 숫자를 적어도 전부 문자열 형태로 넘어가기 때문에 query string
- fragment
    - html 내부 북마크 용도로 사용
    - `서버로 전송되는 정보는 아님`

## 웹 브라우저의 요청 흐름

`https://google.com/search?q=hello&hl=ko`  
웹 브라우저에 이 URL을 치고 엔터를 누르면 어떤 일이 일어날까?

1. DNS에서 `google.com` 도메인 네임을 조회 -> IP 주소로 변환
2. 포트가 생략됐는데, `https` 프로토콜이므로 `443`
3. 브라우저가 `HTTP 요청 메시지` 생성
    - `GET /search?q=hello&hl=ko HTTP/1.1 Host: www.google.com`
4. SOCKET 라이브러리를 통해 요청 메시지가 OS로 전달됨
5. TCP/IP 연결 (IP, PORT)
6. `HTTP 요청 메시지`를 포함하는 `TCP/IP 패킷` 생성
    - ( 출발지 IP, PORT, 도착지 IP, PORT, HTTP 요청 메시지 ) 정보들을 담고 있음
7. 인터넷에 `TCP/IP 패킷`을 던짐
8. 수많은 인터넷 노드를 거쳐 구글 서버에 패킷이 도착함
9. 구글 서버는 `TCP/IP 패킷`을 뜯어내고, 안에 들어있던 `HTTP 요청 메시지`를 꺼냄
10. `HTTP 요청 메시지`를 해석 및 처리하고, `HTTP 응답 메시지`를 만들어냄
    - `HTTP/1.1 200 OK Content-Type: text/html; charset=UTF-8 Content-Length: 3423 <html><body>...</body></html>`
11. 구글 서버에서도 응답 메시지를 감싸는 `TCP/IP 패킷`을 만듦
    - 마찬가지로 ( 출발지 IP 및 PORT, 도착지 IP 및 PORT, HTTP 응답 메시지 ) 정보들을 담고 있음
12. 웹 브라우저에서 `TCP/IP 패킷`을 뜯어내고, 안에 들어있던 `HTTP 응답 메시지`를 꺼냄
13. 꺼낸 HTML 데이터를 브라우저에 렌더링(표시)

## HTTP

HTTP: `HyperText Transfer Protocol`  
하이퍼텍스트 파일, 즉 HTML 파일을 전송하기 위해 만들어진 프로토콜

- 모든 것이 HTTP다!
    - 최근에는 거의 모든 데이터를 HTTP 프로토콜에 담아서 전송하고 있다
    - HTML, 텍스트 파일
    - 사진(이미지), 음성, 영상, 파일
    - 서버 통신에 사용되는 JSON, XML
    - ...
    - 거의 모든 형태의 데이터를 전송할 수 있다
    - 심지어는 서버끼리 데이터를 주고받을 때도 그냥 이 HTTP 프로토콜을 쓴다!
    - `지금은 HTTP 시대`
        - 게임 서버처럼 특수한 경우를 제외하고는 `TCP`를 직접 연결해서 사용하지 않는다. HTTP 프로토콜을 사용해 통신함

- HTTP의 역사
    - HTTP/0.9
        - 1991년, `GET` 메서드만 지원
        - HTTP 헤더 X
    - HTTP/1.0
        - 1996년, 다양한 메서드 지원 추가
        - HTTP 헤더 추가
    - HTTP/1.1
        - 1997년, `가장 많이 사용`
        - `RFC2068` -> `RFC2616` -> `RFC7230 - RFC7235`
        - 2014년에 개정된 최신 스펙인 `RFC7230 - RFC7235` 버전을 봐야 한다!
    - HTTP/2
        - 2015년, 성능 개선
    - HTTP/3
        - 진행중
        - TCP 대신 UDP를 베이스 프로토콜로 사용, 성능 개선

> `HTTP/1.1` 에 대해 잘 아는게 중요하다! 이 강의에서도 HTTP/1.1 위주로 설명할 것

## 클라이언트-서버 구조

HTTP는 `클라이언트-서버` 구조로 되어 있다  
- `Request - Response` 구조
- 클라이언트는 서버에 요청을 보내고, 응답을 기다린다
- 서버는 요청에 대한 결과를 만들어 응답한다

## 무상태 프로토콜 지향

`Stateless` ?
- 서버가 클라이언트의 상태를 보존하지 않는다
    - 장점: 서버를 `쉽게 확장`할 수 있다 (`스케일 아웃`, 수평 확장이 가능하다)
    - 단점: 클라이언트가 서버로 `추가적인 데이터를 전송`해야 한다

```
- 상태를 유지하는 경우

    - 고객: 이 `노트북` 얼마인가요?
    - 점원 A: 100만원 입니다.

    - 고객: `2개` 구매하겠습니다.
    - 점원 A: 200만원입니다. 신용카드, 현금 중 어떤걸로 구매하시나요?

    - 고객: `신용카드`로 구매하겠습니다.
    - 점원 A: 200만원 결제 완료됐습니다.

- 상태를 유지하는데, 만약 중간에 점원이 바뀐다면?

    - 고객: 이 `노트북` 얼마인가요?
    - 점원 A: 100만원 입니다.

    - 고객: `2개` 구매하겠습니다.
    - 점원 B: ?? 무엇을 2개 구매하시나요?

    - 고객: `신용카드`로 구매하겠습니다.
    - 점원 C: ??? 무엇을 몇 개 신용카드로 구매하시나요?

- 무상태(Stateless)일 경우

    - 고객: 이 `노트북` 얼마인가요?
    - 점원: 100만원 입니다.

    - 고객: `노트북 2개` 구매하겠습니다.
    - 점원: 노트북 2개는 200만원입니다. 신용카드, 현금 중 어떤걸로 구매하시나요?

    - 고객: `노트북 2개를 신용카드로 구매`하겠습니다.
    - 점원: 200만원 결제 완료됐습니다.

- 무상태일 때 중간에 점원이 바뀐다면? <- 고객이 요청 처리에 필요한 정보를 모두 보내고 있기 때문에 문제가 발생하지 않는다

    - 고객: 이 `노트북` 얼마인가요?
    - 점원 A: 100만원 입니다.

    - 고객: `노트북 2개` 구매하겠습니다.
    - 점원 B: 노트북 2개는 200만원입니다. 신용카드, 현금 중 어떤걸로 구매하시나요?

    - 고객: `노트북 2개를 신용카드로 구매`하겠습니다.
    - 점원 C: 200만원 결제 완료됐습니다.
```

- Stateless 방식도 실무에서 사용하다 보면 마주치는 `한계`가 있다
    - 모든것을 무상태로 설계할 수 있는 경우도 있지만, `무상태로 설계할 수 없는 경우`도 있다
        - 로그인이 필요하지 않은 단순 서비스 소개 화면은 무상태로 설계할 수 있다
        - `로그인` 기능은 상태를 유지하게 된다
            - 일반적으로, 브라우저 쿠키와 서버 세션등을 활용해 상태를 유지하도록 구현한다
    - 보내야 하는 데이터 양이 많아진다

> 웹 애플리케이션을 설계할 땐, `최대한 무상태로 설계`하고,  
> 어쩔 수 없이 `상태 유지가 필요할 경우`에는 `최소한으로 사용`

> 대용량 트래픽에 대응하는 방법: 서버에서 최대한 상태를 유지하지 않도록 설계하기!

## 비연결성

- HTTP는 기본적으로 `연결을 유지하지 않는 모델`이다
- 일반적인 경우, `초 단위 이하의 빠른 속도로 응답`한다
    - 한시간동안 수천 명이 서비스를 사용한다 하더라도, 실제로 서버에서 `동시에 처리하는 요청의 수`는 수십개 이하로 매우 적다
    - 즉, `서버 자원을 효율적으로 사용`할 수 있다

- HTTP의 비연결성으로 인한 한계점: 유저가 서버에 다양한 자원을 요청하는 경우, 매번 TCP의 3-way handshake 과정을 통해 `새로 연결을 맺어야 한다`
    - 최근에는 HTTP 지속 연결(Persistent Connections, Keep-Alive)을 도입해서 극복함
    - 클라이언트에서 필요한 리소스들을 모두 받을때까지 클라이언트와의 연결을 유지하는 것

## HTTP 메시지

- HTTP 메시지의 구조
    - `start-line`
        - 요청 메시지일 경우 `request-line`
            - `method`(space)`request-target`(space)`HTTP-version`
            - `HTTP 메서드`(공백)`리소스 path`(공백)`HTTP 버전`
        - 응답 메시지일 경우 `status-line`
            - `HTTP-version`(space)`status-code`(space)`reason-phase`
            - `HTTP 버전`(공백)`상태 코드`(공백)`상태 코드에 대한 설명`
    - `header`
        - `field-name:`(OWS)`field-value`(OWS)
            - OWS: 공백을 허용 (띄워도 되고, 안 띄워도 된다)
            - `Host:www.google.com` 가능
            - `Host: www.google.com` 가능
            - `Host :www.google.com` 불가능
            - `Host : www.google.com` 불가능
            > `field-name` 다음에는 공백 없이 일단 콜론이 와야 한다!
            > `field-value` 전에 공백을 넣을지 말지는 선택 가능
        - HTTP 헤더의 역할, 용도
            - HTTP 전송을 위해 필요한 모든 부가정보들을 담고 있다!
                - 메시지 바디의 데이터 형식 (text/html)
                - 메시지 바디의 크기 (3423 Byte)
                - 압축 여부
                - 인증 정보
                - 요청 클라이언트의 브라우저 정보 (브라우저 에이전트)
                - 서버 측 애플리케이션에 대한 정보
                - 캐시 관리 정보
                - ...
    - `empty line` - CRLF, 필수!
    - `message body` - 선택
        - 실제로 전송할 데이터 (HTTP 헤더에는 메타데이터만 있다)
        - HTML 문서, 이미지, 영상, JSON, ...
        - `Byte`로 표현할 수 있는 모든 데이터 전송 가능!

## HTTP API 및 URI 설계

- API URI를 어떻게 설계할 것인가?
- 단순하게 생각해보자면..
    - 회원목록 조회: `/read-member-list`
    - 회원 조회: `/read-member-by-id`
    - 회원 등록: `/create-member`
    - 회원 수정: `/update-member`
    - 회원 삭제: `/delete-member`

> 위 URI들은 좋은 URI 설계일까??
>
> URI에서 가장 중요한 것은 `리소스 식별`이다!!!!!
> URI는 `리소스의 의미`를 담고 있어야 한다

- 그럼 리소스의 의미는 뭘까?
    - 회원을 `등록`하고, `수정`하고, `조회`하는건 `리소스가 아니다!!`
    - `회원이라는 개념 자체가 바로 리소스다`
- 리소스를 어떻게 식별하는게 좋을까?
    - 등록, 수정, 조회 같은 `동작은 배제`
    - `회원이라는 리소스를 URI에 매핑`

## HTTP 메서드

HTTP 메서드는 `클라이언트가 서버에 요청할 때 기대하는 행동`이다
- `GET`은 데이터를 달라는 것
    - 서버의 리소스 조회
    - 서버에 전달하고 싶은 추가적인 데이터는 query를 통해 전달
    - 최근 스펙에서는 GET 요청도 메시지 바디에 데이터를 담을 수 있지만, 아직 지원하지 않는 서버가 많다 (바디에 데이터 안 담는걸 권장)
- `POST`는 데이터를 줄 테니 등록하거나 처리해달라는 것
    - 서버로 보내는 요청 데이터 처리
    - `메시지 바디를 통해 서버로 요청 데이터를 전달`
    - 서버는 전달받은 데이터를 처리
    - `POST 요청 데이터를 어떻게 처리할지`는 `리소스마다 따로 정해둬야 한다!`
        - 주로 신규 리소스를 등록하는데 사용
        - `프로세스 상태를 변경하는데도 사용!`
            - `POST /orders/{orderId}/start-delivery`
            - 주문의 상태를 `배달 시작`으로 변경하는 프로세스를 수행하기 위한 URI
            - 이런 URI를 `컨트롤 URI`라고 한다
            - 새로운 리소스를 생성하는 것이 아니라, 기존 리소스(주문)의 상태가 변경됨
    - `다른 HTTP 메서드로 처리하기 애매`한 경우에도 POST를 사용한다. `애매하면 POST`
- `PUT`은 리소스를 대체해달라는 것
    - 파일을 폴더에 넣는 것과 비슷
    - 폴더 안에 이름이 겹치는 파일이 없으면 추가됨
    - 이름이 겹치는 파일이 있을 경우 덮어씀
    - ***중요***  `클라이언트가 리소스를 식별!`
        - 클라이언트가 정확한 리소스 위치를 알고, URI를 지정할 수 있어야 한다
- `PATCH`는 리소스를 부분적으로 변경해달라는 것
    - 리소스의 일부만 바꿔달라는 것
    - 회원의 이름을 변경하는것처럼 특정 필드의 값을 변경
- `DELETE`는 리소스를 삭제해달라는 것

## HTTP 메서드의 속성

- 안전
    - 리소스를 변경하지 않는다
    - `GET`, `HEAD`는 안전함
    - `PUT`, `POST`, `DELETE`는 안전하지 않음
- 멱등
    - 한 번 호출하던, 두 번 호출하던, 몇 번을 호출하던 결과가 똑같다
    - `GET`, `PUT`, `DELETE`는 멱등
    - `POST`는 `멱등이 아니다!!`
        - 두 번 호출하게 되면 같은 결제가 중복으로 발생할 수도 있다
    - 멱등한 HTTP 메서드의 경우, 자동 복구 메커니즘을 활용할 수 있다
        - DELETE 요청을 날렸는데 서버에서 확인 응답이 없다면, DELETE 요청을 한번 더 보내도 상관없다
- 캐시 가능
    - 응답 결과로 받은 리소스를 캐싱해둬도 되는가?
    - `GET`, `HEAD`, `POST`, `PATCH`는 캐시 가능
        - 하지만 실제로는 `GET`, `HEAD` 정도만 캐싱한다
        - `POST`, `PATCH`의 경우 HTTP 메시지 바디 영역까지 캐시 키로 고려해야 하는데, 구현이 쉽지 않다

## 클라이언트에서 서버로 데이터를 보내는 방법

크게 두가지 방법이 있다
1. `쿼리 파라미터`를 통한 데이터 전달
    - GET
    - 주로 정렬 필터에 사용 (검색어)
2. `HTTP 메시지 바디 영역`을 통한 데이터 전달
    - POST
    - PUT
    - PATCH
    - 회원가입, 상품 주문, 리소스 등록, 리소스 변경 등에 사용

크게 네가지 상황이 있다
1. 정적 데이터 조회
    - 이미지, 정적 텍스트 문서 조회 등에 사용
    - 추가적인 데이터 전달이 필요하지 않은 경우가 대부분이다
    - GET 메서드를 사용하고, 일반적으로 쿼리 파라미터 없이 리소스 경로만으로도 단순하게 조회 가능하다
2. 동적 데이터 조회
    - 조회 조건을 좁혀주는 필터(검색어), 조회 결과를 정렬하는 정렬 조건 등에 주로 사용
    - GET 메서드를 사용하고, 추가적인 데이터는 쿼리 파라미터를 통해 전달한다
3. HTML Form을 통한 데이터 전송
    - 회원가입, 상품 주문, 데이터 변경 등에 사용
    - `Content-Type: application/x-www-form-urlencoded` 사용
        - HTML 폼에 입력된 데이터를 POST 메시지 바디 영역에 담아서 전송 (key=value&key=value 형식, 쿼리 파라미터와 같은 형태)
        - 전송 데이터를 URL 인코딩 처리한다
            - `abc김` -> `abc%EA%B9%80`
    - `Content-Type: multipart/form-data` 를 사용하기도 한다
        - 파일 업로드 기능처럼 바이너리 데이터를 다뤄야 하는 경우 사용
        - 서로 다른 종류의 여러 파일들과 폼에 입력된 내용을 함께 보낼 수 있다 (그래서 이름이 multipart)
4. HTTP API를 통한 데이터 전송
    - GET, POST, PUT, PATCH, DELETE 등 다양한 HTTP 메서드를 모두 활용한다
    - `Content-Type: application/json`을 주로 사용한다 (de facto standard, 사실상 표준)

## HTTP API 설계 예시

### 회원 관리 시스템

- 회원 목록 조회: `/members`, `GET`
- 회원 등록: `/members`,  `POST`
- 회원 조회: `/members/{id}`, `GET`
- 회원 수정: `/members/{id}`, `PATCH`, `PUT`, `POST`
    - 수정은 조금 고민을 해봐야 함
    - `PUT`은 기존 리소스를 완전히 덮어쓰는 것
    - 클라이언트에서 완전히 덮어써도 문제가 없는 상황이라면 `PUT`을 써도 된다
        - 하지만 그런 상황은 생각보다 많지 않을 것
        - 클라이언트가 필요한 정보를 하나도 빠뜨리지 않고 전부 보내줘야 하기 때문
        - 그래서 회원 정보를 수정할때는 `PUT`보다 `PATCH`를 쓰는게 좋다
    - 게시물 수정 같은 경우 `PUT`을 써도 됨
    - 이것도 저것도 아니고 애매하다? `POST`를 써라
- 회원 삭제: `/members/{id}`, `DELETE`

> 애매하면 POST!!

### 파일 관리 시스템

- 파일 목록 조회: `/files`, `GET`
- 파일 조회: `/files/{filename}`, `GET`
- 파일 등록: `/files/{filename}`, `PUT`
- 파일 삭제: `/files/{filename}`, `DELETE`
- 파일 대량 등록: `/files`, `POST`
    - `PUT /files/{filename}`을 사용해서 파일을 등록하고 있다
    - 그래서 `POST`의 의미를 내가 임의로 정할 수 있다!

### 참고하면 좋은 URI 설계 개념
> https://restfulapi.net/resource-naming

- 문서(document)
    - 단일 개념
    - 파일 하나, 객체 인스턴스, 데이터베이스 레코드, ...
- 컬렉션(collection)
    - 서버가 관리하는 리소스 디렉토리
    - 서버가 URI를 관리
    - `실무에서는 대부분 컬렉션을 사용`
- 스토어(store)
    - 클라이언트가 관리하는 자원 저장소
    - 클라이언트가 URI를 관리
- 컨트롤러(controller), 컨트롤 URI
    - 문서, 컬렉션, 스토어만 가지고 해결하기 어려운 추가 프로세스 실행
    - 동사를 사용
    - `/orders/{orderId}/start-delivery`

> 기본적으로 `URI는 리소스만을 표현하도록 설계`
> `컬렉션`과 `GET`, `POST`, `PUT`, `PATCH`, `DELETE` 를 가지고 최대한 해결해본다
>
> 그러다 리소스를 나타내는 URI만으로는 `해결이 안 되는 상황`이 생긴다면 `컨트롤 URI 사용`을 고려

---

`POST`로 리소스를 `등록`하는 경우, 클라이언트에서는 리소스를 추가할 `컬렉션만 지정`해서 요청을 보내고,  
`서버`에서 요청을 처리한 다음 `리소스 URI를 결정하고 만들어준다!`

- `POST`로 컬렉션에 데이터 추가 요청
    - 컬렉션?
        - 서버가 관리하는 리소스 디렉토리
        - 서버가 리소스의 URI를 생성하고 관리
        - 이 예시에서는 `members`가 컬렉션
    - `클라이언트는 등록될 리소스의 URI를 모른다`
```http
POST /members HTTP/1.1
Content-Type: application/json

{
    "username": "doorcs",
    "age": 27
}
```

요청을 받은 서버에서 데이터를 처리하고, `신규 리소스 식별자를 생성` (`/members/100`)

- 응답 예시
```http
HTTP/1.1 201 Created
Content-Type: application/json
Content-Length: 34
Location: /members/100

{
    "username": "doorcs",
    "age": 27
}
```

> POST로 등록할 땐, 서버에서 리소스 URI를 결정하고 만들어준다

---

`PUT` 으로 리소스를 `등록`하는 경우, `클라이언트가 리소스 URI를 알고 있어야 한다!`

- `PUT`으로 스토어에 리소스 등록 요청
    - 스토어?
        - 클라이언트가 관리하는 리소스 저장소
        - 클라이언트가 리소스의 URI를 알고 관리
        - 이 예시에서는 `/files`가 스토어
    - `클라이언트가 직접 리소스의 URI를 지정한다!`
```http
PUT /files/star.png HTTP/1.1
Content-Type: image/png
Content-Length: 2048

{바이너리 데이터}
```

- 응답 예시
    - 신규 등록이 아니고, 데이터가 이미 존재할 경우 `200 OK`나 `204 No Content`를 응답하도록 구현하기도 한다
```http
HTTP/1.1 201 Created
Content-Type: image/png
Content-Length: 2048
Location: /files/star.png

```

## HTTP 상태 코드

클라이언트가 보낸 요청의 처리 상태를 응답에서 알려주는 기능

- `1xx` - Informational - 요청이 수신되어 처리중
    - 이건 거의 사용하지 않는다. 실무에서 한번도 본 적 없다고 함
- `2xx` - Successful - 요청이 정상적으로 처리됨 (성공)
    - `200 OK`
        - 요청 성공
    - `201 Created`
        - 요청에 성공해서 새로운 리소스가 생성됨
        - 생성된 리소스는 `응답 HTTP 헤더의 Location 필드`로 식별
    - `202 Accepted`
        - 요청이 접수되었으나 처리가 완료되지는 않았음
        - `배치 처리` 같은곳에서 사용
    - `204 No Content`
        - 서버가 요청을 성공적으로 처리했지만, `응답 HTTP 바디에 보낼 데이터가 없음`
            - 웹 문서 편집기에서 저장 버튼을 누르는 경우, 저장에 성공했다는 사실만 알 수 있으면 된다
- `3xx` - Redirection - 요청을 완료하려면 추가 행동이 필요함
    - 웹 브라우저는 `3xx 응답의 HTTP 헤더`에 `Location 헤더`가 있으면 `해당 Location으로 자동 이동`한다!
    - 리다이렉션의 종류
        - 영구 리다이렉션 - 특정 리소스의 URI가 영구적으로 변경된 경우
            - `/event` 리소스의 위치가 `/new-event`로 변경된 경우
            - 검색 엔진 등에서 URL 변경을 반영하기도 함
        - 일시 리다이렉션 - 일시적인 변경
            - 주문 완료 후, 주문 내역 화면으로 이동
            - `PRG 패턴` : Post -> Redirect -> Get
                - POST로 주문한 직후에 웹 브라우저를 새로고침한다면? 중복 주문이 될 수도 있다 (물론 서버에서 막는 처리를 하긴 하지만)
                - PRG 패턴을 적용하면 이런 문제를 막으면서 사용성도 좋아진다
                    - POST에 대한 응답으로 `302 Found`를 보내준다
                    - 클라이언트는 자동으로 리다이렉트된다
                    - 즉, 결과 페이지로 GET 요청을 보내는 것이기 때문에 몇번이고 새로고침하더라도 문제가 발생하지 않는다!
        - 특수 리다이렉션 - `결과 대신 캐시를 사용해라`
    - `301 Moved Permanently`
        - 리다이렉트시 `요청 메서드가 GET으로 변하고, HTTP 바디 내용이 제거`될 수도 있다
        - 대부분의 구현에서 GET으로 변하고 바디 내용이 날라감
    - `308 Permanent Redirect`
        - 리다이렉트시 `요청 메서드와 HTTP 바디 내용을 유지`한다
    - `302 Found`
        - 리다이렉트시 `요청 메서드가 GET으로 변하고, HTTP 바디 내용이 제거`될 수도 있다
        - 대부분의 구현에서 GET으로 변하고 바디 내용이 날라감
    - `307 Temporary Redirect`
        - 리다이렉트시 `요청 메서드와 HTTP 바디 내용을 유지`한다
    - `303 See Other`
        - 리다이렉트시 `요청 메서드가 GET으로 변한다`
        - 302 Found는 동작이 명확하지 않기 때문에, 명확하게 요청 메서드를 GET으로 변경하고 싶을 때 사용
    - `304 Not Modified`
        - 이건 진짜 많이 쓴다! 캐시를 목적으로 사용함
        - 클라이언트에게 리소스가 수정되지 않았음을 알려준다
            - 따라서 클라이언트는 로컬 환경에 저장된 캐시를 재사용한다
            - 캐시로 리다이렉트 하는 셈이다
        - `304 응답은 HTTP 바디를 포함하면 안 된다`
- `4xx` - Client Error - 클라이언트 오류, 잘못된 요청을 보냈기 때문에 서버가 요청을 수행할 수 없음
    - 클라이언트의 요청 데이터가 잘못됐다는 의미
        - 클라이언트는 요청 내용을 다시 검토해봐야 한다
        - 요청 파라미터가 잘못되거나, API 스펙에 맞지 않을 때
    - 서버 개발자라면 Validation을 철저히 수행해서 잘못된 요청에 대해 400번대 응답을 내려줘야 한다!!
        - 요청에 문제가 있는데 500번대 응답을 내려주면 안된다..!
    - `401 Unauthorized`
        - 오류 메시지는 Unauthorized이지만.. `인증(Authentication)되지 않았다는 의미`
            - 인증(Authentication) - 본인이 누구인지 확인 (로그인 여부)
            - 인가(Authorization) - 권한이 적절한지 확인 (접근 권한 여부)
        - 401 오류 발생시, 응답에 `WWW-Authenticate` 헤더와 함께 인증 방법을 설명
    - `403 Forbidden`
        - 서버가 요청을 이해했지만 수행을 거부함
        - 인증은 됐지만, `권한이 없는 경우`
            - 일반 사용자가 관리자만 조회할 수 있는 리소스에 접근하는 경우
    - `404 Not Found`
        - 요청 리소스를 찾을 수 없음
        - 또는, `클라이언트가 권한 없는 리소스에 접근할 때 해당 리소스의 존재 여부를 숨기고 싶을 때`도 사용
        - 이게 왜 클라이언트 오류냐? 서버 입장에서는 존재하지 않는 리소스인데 클라이언트가 요청한 것이기 때문
- `5xx` - Server Error - 서버 오류, 서버에 문제가 있어서 요청을 수행할 수 없음
    - 서버 문제로 인한 것이기 때문에, 동일한 요청을 재시도하면 성공할 수도 있다
    - `500 Internal Service Error`
        - 서버 내부 문제로 오류 발생
        - `애매하면 500 오류`
    - `504 Service Unavailable`
        - 서버가 일시적인 과부하 또는 예정된 작업으로 인해 잠시동안 요청을 처리할 수 없다
        - `Retry-After` 헤더를 사용해서 얼마 뒤에 복구될 예정인지 알려줄 수도 있음

만약 모르는 상태 코드가 나타난다면?
- 클라이언트가 인식할 수 없는 상태 코드를 서버가 내려주면?
    - 클라이언트는 상위 레벨의 상태코드로 해석해서 처리한다
        - 299 -> 2xx (Successful)
        - 451 -> 4xx (Client Error)
        - 599 -> 5xx (Server Error)
    - 이렇게 하면, 미래에 새로운 상태 코드가 추가되더라도 클라이언트를 변경하지 않아도 됨

## HTTP 헤더

> 2014년에 `RFC7230 - RFC7235`로 개정되면서 많이 달라졌다!

`표현` 헤더
- 표현 헤더는 `HTTP 요청`과 `HTTP 응답`에서 모두 사용한다
- Content-Type : 표현 데이터의 형식
    - 미디어 타입, 문자 인코딩
    - `Content-Type: text/html;charset=UTF-8`
    - `Content-Type: application/json`
    - `Content-Type: image/png`
- Content-Encoding : 표현 데이터의 압축 방식
    - 표현 데이터를 압축하기 위해 사용
    - 데이터를 보내는 쪽에서 압축 후 인코딩 헤더를 추가
    - 데이터를 읽는 쪽에서 인코딩 헤더의 정보를 활용해 압축 해제
    - `Content-Encoding: gzip`
    - `Content-Encoding: deflate`
    - `Content-Encoding: identity` - 이건 압축을 안 했다는 의미
- Content-Language : 표현 데이터의 자연 언어
    - `Content-Language: ko-KR`
    - `Content-Language: en-US`
- Content-Length : 표현 데이터의 길이
    - 바이트(Byte) 단위를 사용
    - `Transfer-Encoding`, 분할 전송을 사용할 때는 Content-Length 헤더를 사용하면 안 됨

`협상` 헤더
- 클라이언트의 선호를 서버에 알려주는 것
    - `HTTP 요청` 에서만 사용된다
- Accept : 클라이언트가 선호하는 미디어 타입 전달
- Accept-Charset : 클라이언트가 선호하는 문자 인코딩
- Accept-Encoding : 클라이언트가 선호하는 압축 인코딩
- Accept-Language : 클라이언트가 선호하는 자연 언어
    - `Accept-Language: ko-KR` 헤더를 포함한 요청을 받았는데 서버에서 한국어도 지원한다면 한국어로 응답을 내려줄 수 있다
        - 독일어가 기본값이고 영어도 지원하는 서버라면? `ko-KR`에 매칭되는게 없으니 기본값인 독일어로 응답할 것 (차라리 영어가 나은데도)

협상 헤더에는 `우선순위`를 설정할 수 있다
- `Quality Values`, `q값` 사용
    - 0에서 1 사이의 값, `값을 생략하면 1`
- `q값`이 큰 것이 우선
    - `Accept-Language: ko-KR,ko;q=0.9,en-US;q=0.8;en;q=0.7`
        - 우선순위 1번: ko-KR (q=1)
        - 우선순위 2번: ko (q=0.9)
        - 우선순위 3번: en-US (q=0.8)
        - 우선순위 4번: en (q=0.7)
- `더 구체적인 값`이 우선
    - `Accept: text/*,text/plain,text/plain;format=flowed,*/*`
        - 우선순위 1번: text/plain;format=flowed
        - 우선순위 2번: text/plain
        - 우선순위 3번: text/*
        - 우선순위 4번: \*/\*

`전송 방식` 헤더
- 단순 전송
    - `Content-Length: 3423`
- 압축 전송
    - `Content-Encoding: gzip`
    - `Content-Length: 521`
- 분할 전송
    - `Transfer-Encoding: chunked`
    - `Content-Length`를 넣으면 안 된다!
        - 여러 덩어리(chunk)로 나눠서 보내고, 각각의 덩어리가 길이 정보를 가지고 있기 때문
- 범위 전송
    - `Range: bytes=1001-2000` (클라이언트 요청)
    - `Content-Range: bytes 1001-2000 / 2000` (서버 응답)

`일반 정보` 헤더 (정보성 헤더)
- From
    - 유저 에이전트의 이메일 정보
    - 일반적으로 잘 사용되지 않는다
    - 요청에서 사용
- Referer
    - 현재 요청된 페이지의 `이전 웹 페이지 주소`
    - A 페이지에서 B 페이지로 이동하는 경우, B 페이지를 요청할 때 `Referer: A` 헤더를 포함해서 요청한다
    - Referer를 통해 `페이지 유입 경로`를 분석할 수 있다
    - 요청에서 사용
    - `referer`는 사실 `referrer` 라는 단어의 오타다
        - 이미 인터넷에서, 많은 서버들에서 이걸 사용하고 있기 때문에 고치지 않고 그대로 사용하기로 결정..
- User-Agent
    - 유저 에이전트의 애플리케이션 정보 (어떤 브라우저를 사용중인지 등등)
    - 통계 정보를 뽑기 좋다 (사용자들이 어떤 브라우저에서 많이 들어오는구나)
    - 어떤 종류의 브라우저에서 장애가 발생하는지 파악할 수 있다
    - 요청에서 사용
- Server
    - 요청을 실제로 처리하는 ORIGIN 서버의 소프트웨어 정보
    - `Server: Apache/2.2.22 (Debian)`
    - 응답에서 사용
- Date
    - 메시지가 발생한 날짜와 시간 정보
    - `Date: Tue, 15 Nov 1994 08:12:31 GMT`
    - 응답에서 사용

`특별한 정보` 헤더
- Host
    - 요청에서 사용
    - 요청한 호스트 정보 (도메인)
    - `필수 헤더!!`
- Location
    - 응답에서 사용
    - 페이지 리다이렉션
    - 웹 브라우저는 300번대 응답에 Location 헤더가 있으면, Location에 적힌 URI로 자동 리다이렉트된다
    - `201 Created` 응답도 Location 헤더를 사용하지만, 이 경우에는 요청에 의해 생성된 리소스의 URI를 의미함 (리다이렉트 용도가 아님)
- Allow
    - 응답에서 사용
    - 서버에서 허용해둔 HTTP 메서드 종류
    - `405 Method Not Allowed`를 내려줄 때, 응답 헤더에 Allow 정보를 포함해야 함
        - Allow: GET, HEAD, PUT
    - 자주 사용하지는 않는다
- Retry-After
    - 응답에서 사용
    - 유저 에이전트가 다음 요청을 보내기까지 기다려야 하는 시간
    - `503 Service Unavailable`을 내려줄 때, 언제까지 서비스 이용이 불가능한지 알려줄 수 있음
        - `Retry-After: Fri 31, Dec 1999 23:59:59 GMT` (날짜 표기)
        - `Retry-After: 120` (초 단위 표기)

`인증` 헤더
- Authorization
    - 요청에서 사용
    - 클라이언트의 인증 정보를 서버에 전달
    - `Authoriaztion: Bearer xxxxxxxxxxxxxx`
- WWW-Authenticate
    - 응답에서 사용
    - 리소스 접근 시 필요한 인증 방법을 전달
    - `401 Unauthorized` 응답과 함께 사용
    - `WWW-Authenticate: Bearer scope="" error="invalid_token" error_description="the token does not have a valid audience"`

HTTP 프로토콜은 기본적으로 Stateless  
상태를 가지지 않기 때문에 사용자가 요청을 처리하기 위해 필요한 데이터를 매번 함께 보내줘야 한다  
그렇다고 유저 정보를 매번 쿼리 파라미터에 같이 담아서 보내게 되면? `심각한 보안 문제`

HTTP의 무상태성에서 오는 단점을 극복하기 위해 `쿠키`라는 개념을 도입
`사용자의 모든 HTTP 요청에 쿠키 정보가 자동으로 포함된다`

`쿠키` 헤더
- Set-Cookie
    - 서버에서 클라이언트로 쿠키를 전달하기 위해 사용하는 헤더
    - 쿠키의 사용처?
        - 사용자의 로그인 세션을 관리
        - 광고 정보 트래킹
    - 쿠키에 저장된 정보는 `항상 서버에 같이 전송`된다
        - `추가적인 네트워크 트래픽을 유발`한다
        - 그래서 최소한의 정보만을 사용해야 함! (세션 id, 인증용 JWT 토큰 등)
    - 서버에 전송하지 않고, 웹 브라우저 내부에 데이터를 저장해두고 싶으면 웹 스토리지를 사용하기 (`localStorage`, `sessionStorage`)
    - `쿠키든 웹 스토리지든 보안에 민감한 정보는 절대 저장하면 안된다!!!` - 주민번호, 신용카드 번호 등등..
    - `Set-Cookie: sessionId=abcde1234;expires=Sat, 26-Dec-2020 00:00:00 GMT;path=/;domain=google.com;Secure`
        - 쿠키에는 `세션 쿠키`와 `영속 쿠키`가 있다
            - 세션 쿠키: 만료 정보가 없는 쿠키. 브라우저 종료 시 까지만 유지된다
            - 영속 쿠키: 만료 정보가 있는 쿠키. 브라우저 종료 여부와 무관하게 만료 전까지 유지된다
                - `expires=Sat, 26-Dec-2020 04:39:21 GMT`
                    - 만료일이 되면 쿠키를 삭제한다
                - `max-age=3600`
                    - 초 단위로 설정된 유효기간이 지나면 쿠키를 삭제한다
                    - 0이나 음수를 지정하면 해당 쿠키를 즉시 삭제할 수 있다
        - 쿠키의 도메인
            - `domain=example.org`
                - 도메인을 명시할 경우, `해당 도메인 + 해당 도메인의 모든 서브도메인`에 적용
                    - `example.org` 뿐만 아니라 하위 도메인인 `dev.example.org` 에 요청을 보낼때도 해당 쿠키를 전송한다
                - 도메인을 생략할 경우, 현재 문서의 도메인에만 적용된다
                    - `example.org`에서 쿠키를 생성하는데 도메인을 생략할 경우, dev.example.org에 요청을 보낼 땐 그 쿠키를 전송하지 않는다
        - 쿠키의 경로
            - `path=/home`
                - `지정한 경로를 포함한 하위 경로`로 요청을 보내는 경우에만 쿠키를 전송한다
                    - `/home/level1` 이나 `/home/level1/level2`에 요청을 보낼 때는 쿠키를 전송하지만 `/hello`에는 쿠키를 전송하지 않는다
                - 일반적으로 `path=/` 를 통해 루트 경로로 지정한다
        - 쿠키의 보안 옵션
            - `Secure`
                - 일반적인 쿠키는 http, https 프로토콜을 구분하지 않고 모두 전송됨
                - `Secure` 가 적용된 쿠키는 https 프로토콜에서만 전송됨
            - `HttpOnly`
                - XSS 공격을 방지하기 위한 옵션
                - 자바스크립트에서 해당 쿠키에 접근할 수 없도록 막아줌 (`document.cookie`를 통한 접근 불가)
                - HTTP 요청 전송시에만 사용됨
            - `SameSite`
                - XSRF 공격을 방지하기 위한 옵션
                - 비교적 최근에 추가된 옵션이다
                - 요청 도메인과 쿠키에 설정된 도메인이 같은 경우에만 쿠키를 전송
- Cookie
    - 클라이언트가 서버로 HTTP 요청시 쿠키를 같이 전달하기 위해 사용하는 헤더

네트워크 접근은 `굉장히 비싼 연산`이다. 느리다!!  
데이터가 변경되지 않았는데도 매번 네트워크를 통해 똑같은 데이터를 다운로드받는건 비효율적  
사용자 경험에도 좋지 않다

캐시를 적용하면 `네트워크 사용량 감소`, `브라우저의 로딩 속도 개선`, `좋은 사용자 경험`  
안 쓸 이유가 없다!

`캐시` 헤더
- Cache-Control
    - `Cache-Control: max-age=60`
    - 60초동안은 캐시가 유효하기 때문에, 60초 이내에 동일한 요청이 들어오면 네트워크 접근 없이 그냥 캐시를 꺼내 쓸 수 있다
    - 60초 이후에는 다시 서버에 접근해서 데이터를 다운받고, 캐시 정보를 갱신해야 한다
        - `서버에서 데이터가 변경되지 않았는데도 굳이 새로 다운받을 필요가 있을까?`

`검증` 헤더 (캐시 관련!)
- 캐시가 만료됐다고 하더라도, 서버에서 데이터가 변경되지 않았다면?
    - `데이터를 다시 전송하는 대신, 저장해뒀던 캐시를 재사용하도록 할 수 있다`
    - 대신, 클라이언트의 캐시 데이터와 서버의 데이터가 같다는 사실을 확인할 수 있는 방법이 필요
- Last-Modified
    - 서버에서 응답을 내려줄 때, `리소스의 마지막 수정 시점`을 같이 내려주기 위해 사용하는 헤더
    - `Last-Modified: Fri 31, Dec 1999 23:59:59 GMT`
- If-Modified-Since
    - 클라이언트 측에 캐시된 리소스의 최종 수정 시점을 담는 헤더
    - `If-Modified-Since: Fri 31, Dec 1999 23:59:59 GMT`
    - 서버에서 응답을 내려주기 전에, `서버의 리소스 수정 시점이 If-Modified-Since 이후인지` 확인한다
        - 만약 서버의 리소스가 if-modified-since 이후로 변경되지 않았다면?
        - 실제 데이터 대신 `304 Not Modified` 응답을 내려준다
            - `HTTP 바디가 없는 응답이다!`
            - 클라이언트한테 헤더만 보내주면 되니까 네트워크 트래픽이 확 줄어든다
    - 이런 수정시점 기반 캐싱에는 단점이 있다
        - 1초 미만의 단위(0.x초)로는 캐시 조정이 불가능하다
        - 날짜 기반의 로직을 사용한다
            - 데이터를 수정이 일어나서 최종 수정 시점은 달라졌지만 데이터가 동일할 경우?
        - 서버에서 캐시 로직을 직접 관리할 수 없다
- ETag
    - Entity Tag의 줄임말
    - 서버에서 내려주는 데이터에 `임의의 고유한 버전 태그`를 달아주기 위한 헤더
        - `Etag: "aaaaa"`
    - 클라이언트에서 데이터를 다시 내려받도록 하고 싶을 경우 태그를 변경
        - `Etag: "bbbbb"`
    - 진짜 단순하게, `ETag가 같으면 캐시 유지, 다르면 새로 내려받기!`
    - 캐시 제어 로직을 완전히 서버에서 관리할 수 있다
        - 베타 오픈 기간인 3일동안은 파일 내용이 변경되더라도 ETag를 동일하게 유지
        - 애플리케이션 배포 주기에 맞춰서 ETag 값을 모두 갱신
- If-None-Match
    - 서버에 요청을 보낼 때, `리소스의 ETag 정보`를 함께 보내기 위해 사용하는 헤더
    - `If-None-Match: "aaaaa"`
        - 만약 서버에 저장된 리소스의 ETag가 변경되지 않았다면?
        - 실제 데이터 대신 `304 Not Modified` 응답을 내려줌

`캐시 제어` 헤더
- Cache-Control
    - Cache-Control: max-age
        - 캐시의 유효기간을 설정, 초 단위
    - Cache-Control: no-cache
        - 데이터는 캐시해도 되지만, 항상 오리진(Origin) 서버를 거쳐서 검증하고 사용해라
        - `캐시 서버에서 오리진 서버에 접근할 수 없는 경우, 캐시 서버의 설정에 따라 오류를 발생시키지 않고 캐시 데이터를 내려줄수도 있다!!`
    - Cache-Control: no-store
        - 데이터에 민감한 정보가 있으므로 캐싱하면 안 됨
        - 메모리에서 사용하고 삭제
    - Cache-Control: must-revalidate
        - 캐시 만료 후 최초 조회시 `반드시 오리진 서버에 검증해야함`
        - 오리진 서버 접근에 실패할 경우 `504 Gateway Timeout` 오류를 발생시키도록 스펙에 명시되어 있음
    - Cache-Control: public
        - 응답이 public 캐시에 저장되어도 된다
        - 프록시 캐시에 저장될 수 있다는 의미. 모든 사용자가 공용으로 사용하는 리소스에 사용
    - Cache-Control: private
        - 응답이 해당 사용자만을 위한 것이다 (`기본값`)
    - Cache-Control: s-maxage
        - 프록시 캐시에만 적용되는 max-age
- Pragma
    - HTTP/1.0 하위호환을 위한 헤더
    - Pragma: no-cache
- Expires
    - HTTP/1.0 하위호환을 위한 헤더
    - Expires: Mon, 01 Jan 1990 00:00:00 GMT
        - 캐시 만료일을 정확한 날짜로 지정
        - `지금은 더 유연한 Cache-Control: max-age 사용을 권장!`
            - 해당 헤더와 같이 사용되면 Expires 헤더는 무시됨

> 민감 정보를 다룰 때, 확실하게 캐시를 막으려면?
>
> `Cache-Control: no-cache, no-store, must-revalidate`
> `Pragma: no-cache` - HTTP/1.0 하위호환을 위해 이것까지 넣어주기
